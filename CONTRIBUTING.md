# Contributing to Neksur Documentation

Thank you for helping improve Neksur's documentation. This repository is licensed under **Apache License 2.0** — contributions are permissively licensed and easy to make.

## Quick Process

### Typos, broken links, small fixes

Open a pull request directly. No issue required.

### Substantive changes

Open an issue first describing what you want to change and why. A maintainer will respond within a few days. This avoids wasted effort on changes that don't fit the documentation's structure.

### New pages or sections

Open an issue with a brief outline. Discuss before writing — documentation structure decisions are easier to make on a one-paragraph proposal than on a finished draft.

## Style

- **Audience first.** Each page targets a specific audience (operator, developer, security architect, data platform engineer, AI agent integrator). State the audience in the page intro.
- **Concrete over abstract.** Examples beat prose. Include working code, runnable commands, sample policies.
- **Honest about state.** If a feature is pre-MVP, post-MVP, or behind a feature flag, say so explicitly. Don't pretend the project is further along than it is.
- **Cross-link.** Use relative links between pages. Cross-link concepts pages from reference pages and vice versa.
- **One-sentence paragraphs are fine.** Skim-friendly beats prose-perfect.

Apache 2.0 contributions do not require DCO sign-off (unlike contributions to `neksur-com/neksur` which uses BSL 1.1). Just open a PR.

## Code of Conduct

Participation in this project is governed by the organization-level [`CODE_OF_CONDUCT.md`](https://github.com/neksur-com/.github/blob/main/CODE_OF_CONDUCT.md). Be respectful; we will be too.

## Reporting Security Issues

If a documentation issue describes a security vulnerability in the product itself (not a documentation bug), report it privately per [`SECURITY.md`](https://github.com/neksur-com/neksur/blob/main/SECURITY.md) in the main `neksur` repository — do not open a public issue here.

For documentation-only issues (broken links, outdated examples, factual errors), a public issue is fine.

## Questions

- General questions: open a GitHub issue on this repo with the `question` label
- Architecture / roadmap questions: ask on [`neksur-com/neksur`](https://github.com/neksur-com/neksur)
- Commercial inquiries: `hello@neksur.com`
